## AI Summary
- **What it does**: Splits `auth_expired` into `confirmed` (real 401 / explicit "Session expired" / socket push) vs `unconfirmed` ("session jwt required" / "no backend session token") and adds a corroboration gate тАФ for unconfirmed signals, reads the on-disk token via `auth_get_session_token` up to 3├Ч (300ms apart) before running the destructive `clearSession`. Fixes a boot-race where the identity-flip restart's first RPC fires before the auth profile finishes loading, gets "session jwt required", and the old code blindly cleared the persisted session.
- **Breaking risk**: Low. Plumbs a new `reason` field through the internal `core-rpc-auth-expired` custom event; no public API or Rust change. Existing consumers that don't provide `reason` default to `unconfirmed` (safe path).
- **Security risk**: Low. Confirmed expiries (real 401, socket push) still clear immediately тАФ no weakening of genuine expiry handling. The guard only blocks destruction on an unconfirmed "not loaded yet" signal.
- **Bottom line**: Safe to merge тАФ well-designed fix for a real destructive bug, confirmed+unconfirmed paths correctly separated, tests cover all branches.

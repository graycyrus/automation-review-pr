### What it does

Adds a "Memory Tree" health dashboard to the app's Intelligence/Memory pages. Users see four live tiles — pipeline status (running/paused/syncing/error/idle), last sync time, total chunks stored, and wiki disk size — plus a toggle switch to pause/resume the auto-sync pipeline without restarting the app. Under the hood, two new Rust RPCs (`pipeline_status` and `set_enabled`) aggregate SQLite counters, a disk-size walk, and scheduler-gate state into a single payload. The old half-baked status pills in `Intelligence.tsx` (sourced from `useConsciousItems` + `useMemoryIngestionStatus`) are removed — those showed stale/inaccurate numbers from a legacy pipeline.

This matters because operators previously had zero visibility into whether the Memory Tree was healthy, stalled, or erroring, and no way to pause ingestion without editing config files and restarting.

### Safety & Breaking concerns

**Breaking risk: Low**

Detailed findings:

1. **No security flaws found.** The `set_enabled` RPC only toggles between two known enum values (`Auto`/`Off`) — no string injection vector. The status RPC is read-only. The `walkdir` is scoped to the wiki subdirectory and doesn't follow symlinks. Walkdir errors are caught and logged with redacted paths (no PII leak).

2. **Removed UI elements (header pills).** The old system-status dot + ingestion chip in the Intelligence page header are deleted. These were driven by hooks (`useConsciousItems`, `useMemoryIngestionStatus`, `useIntelligenceStats`) that are **kept** but just no longer called from this page. If any external code relied on those header pills being present, they'd break — but the PR body says the pills showed inaccurate data and the hooks still exist for other consumers. Acceptable tradeoff.

3. **Polling at 1.5s–4s is fine.** The status RPC does two SQL aggregates + a `walkdir` on `spawn_blocking`. For typical wikis this is cheap. Extremely large vaults (hundreds of thousands of files) could make the walkdir expensive at 1.5s cadence, but `saturating_add` + error tolerance prevent crashes — it just gets slow.

4. **Toggle scope is clearly documented.** The 20-min Composio fetch loop is explicitly NOT paused by this toggle (queued for Part 2). This is called out in the PR body, the Rust doc comments, and the TS doc comments. No silent gap.

5. **i18n keys are English-only in non-English locales.** All 12 non-English chunk files got the new keys but with English strings, not translated. This is a common pattern in this repo (translations land later), and `pnpm i18n:check` passes with `missing: 0`. Not a bug, just a UX debt.

6. **Config save error handling is solid.** `set_enabled_rpc` propagates `config.save()` failures as RPC errors, which the frontend surfaces via toast. No silent data loss.

7. **No schema migration needed.** Both RPCs read existing tables (`mem_tree_chunks`, `mem_tree_jobs`) and toggle an existing config enum. No DB changes.

### Bottom line

**Safe to merge** — well-scoped read-only status surface + a single config-enum toggle with proper persistence, hot-reload, idempotency handling, good test coverage (11 Rust + 6 Vitest), and all CI green.

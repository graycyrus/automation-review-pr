## PR Description
Security fix for `is_path_allowed()` symlink bypass (issue #1927). The `browser/image_info.rs` tool was the only filesystem tool that called `is_path_allowed` on the raw path but then read the file without canonicalization + `is_resolved_path_allowed`. A symlink inside the workspace pointing outside it would exfiltrate arbitrary files through this tool.

**Evolution across commits:**
1. `ac05460` — Added `try_canonicalize_under_workspace()` helper to `policy.rs` + 3 symlink tests in `policy_tests.rs`
2. `69ce5d7` — Fixed forbidden_paths regression: skip re-check for paths inside workspace
3. `aa96038` — `cargo fmt` whitespace fix
4. `406f442` — **Reverted all policy.rs changes**. Realized 7/8 filesystem tools already pair `is_path_allowed` + `canonicalize` + `is_resolved_path_allowed`. The real gap was only `image_info.rs`. Added canonicalize + `is_resolved_path_allowed` to `image_info.rs` instead, with a new symlink escape test.

The final diff touches only `image_info.rs`.

### IMPORTANT FINDING — Relative path handling
**file_read.rs** does `workspace_dir.join(path)` before canonicalize. The PR's image_info.rs canonicalizes `path` directly (from `Path::new(path_str)` at line ~159). For relative inputs, this would resolve against the process CWD, not the workspace. However, looking at the image_info.rs code flow: the `path` variable at the canonicalize point is already constructed from the tool input and the tool receives absolute paths (the `is_path_allowed` check at line ~162 already validates the path). The `path` variable is `Path::new(path_str)` where `path_str` comes from the JSON input. **CodeRabbit flagged this as a potential issue** — see dedup section.
